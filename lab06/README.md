# Lab 06 — Jira-Driven Development with MCP

**Duration:** ~75 min  
**SDLC Phase:** Plan → Implement → Validate → Release  
**Autonomy Level:** 🟡 Agent executes, human approves each story transition  
**Prerequisites:** Python 3.10+, VS Code with GitHub Copilot (Agent Mode enabled), a Jira Cloud account with the `SCRUM` project pre-seeded

---

## Learning Objective

Use **Jira MCP** as the collaboration backbone of a real development sprint. You will manage issues, post implementation plans, transition story states, attach evidence, and write a release sign-off — all through Copilot prompts — while implementing the actual code locally using Copilot's coding capabilities.

No external code-repo MCPs are needed. Code is read and edited directly in VS Code. Jira is the single source of truth for planning and delivery.

---

## Use Case

Your team is delivering a **Manager Approval Reminder** capability for an HR workflow system (`FR-001`). Approvals pending more than 48 hours must trigger a reminder with a full audit trail.

The Jira project `SCRUM` already contains the epic and five stories that map directly to five functions you will implement in `reminder_engine.py`.

## Setup


### 2. Configure the Jira MCP server

The `.vscode/mcp.json` already references a **locally patched** Jira MCP server under `.jira-mcp/`. This patch upgrades the deprecated `/rest/api/3/search` endpoint to `/rest/api/3/search/jql` (Jira Cloud API v3) and adds two extra tools: `transition_ticket` and `add_comment`.

When VS Code starts the `jira` server it will prompt for:

| Prompt | Value |
|--------|-------|
| Jira site URL | `https://<your-site>.atlassian.net` |
| Jira account email | your Atlassian account email |
| Jira API token | generate at [id.atlassian.com → Security → API tokens](https://id.atlassian.com/manage-profile/security/api-tokens) |

After filling in credentials, the MCP panel should show **jira — Running** with **17 tools** discovered.

> **Troubleshooting:** If the server shows an error, check the Output panel (MCP channel). Common fixes:
> - Wrong URL format → use `https://yoursite.atlassian.net` (no trailing slash)
> - Expired token → regenerate at Atlassian and restart the server
> - Missing dependencies → run `cd .jira-mcp && npm install --production`

## TDD Loop — Follow This for Every Story

For each story you implement:

1. **Read** the function docstring and TODO comment in `reminder_engine.py`.
2. **Write** the failing test in `tests/test_reminder_engine.py`.
3. **Confirm** the test fails (it should — the function raises `NotImplementedError`).
4. **Implement** the function.
5. **Confirm** the test passes.
6. **Update Jira**: add a completion comment and transition the story to Done.

```bash
# Run all tests at any time
pytest -q tests/test_reminder_engine.py
```

## Step 7 — Implement SCRUM-3: Scheduler

**Functions:** `send_reminder` and `run_scheduler_cycle`

**`send_reminder(approval, now) -> ReminderAttempt`**
- If `manager_email` does not contain `@`: return `ReminderAttempt` with `outcome="FAILED"`, `reason="invalid recipient address"`
- Otherwise: set `approval.last_reminded_at = now`, return `ReminderAttempt` with `outcome="SENT"`, `reason="reminder dispatched"`

**`run_scheduler_cycle(approvals, policy, now=None) -> SchedulerRunResult`**
1. Resolve `run_at = now or utc_now()`
2. Call `validate_policy(policy)`
3. Convert `approvals` to a list
4. Call `find_eligible_approvals(...)` to get eligible items
5. Call `send_reminder(...)` for each eligible approval
6. Return `SchedulerRunResult(evaluated=..., eligible=..., attempts=[...])`

**Write tests, confirm fail, implement, confirm pass:**

```bash
# Expected after implementation: all PASSED
```

## What to Observe

- Every state transition and comment was triggered through **Jira MCP** — no manual board interaction needed.
- Code was read and edited **locally** in VS Code — no code-repo MCP was required.
- The Jira MCP server was **patched in-place** (`.jira-mcp/`) to fix a deprecated API endpoint and add missing tools (`transition_ticket`, `add_comment`), demonstrating that MCP servers are just software you can inspect and extend.
- The evidence JSON provides a traceable artifact linking the Jira epic to the actual scheduler run — a pattern you can reuse for any release gate.

## Jira MCP Tools Reference

| Tool | What it does |
|------|-------------|
| `execute_jql` | Run a JQL query and return matching issues |
| `get_ticket` | Fetch a single issue by key |
| `transition_ticket` | Move an issue to a new status by name |
| `add_comment` | Post a comment to an issue |
| `assign_ticket` | Assign an issue to a user account |
| `create_ticket` | Create a new issue |
| `edit_ticket` | Update summary, description, or labels |
| `list_projects` | List all accessible projects |
| `query_assignable` | List users who can be assigned in a project |
